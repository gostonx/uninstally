# Update System Architecture

Uninstally updates itself with [Sparkle](https://sparkle-project.org) — the
standard, battle-tested update framework for macOS apps distributed outside the
App Store. We deliberately do **not** ship a custom updater.

```
   ┌────────────┐   push tag v1.4.0    ┌─────────────────────┐
   │  Developer │ ───────────────────▶ │   GitHub Actions    │
   └────────────┘                      │  build · ad-hoc     │
                                       │  sign · DMG ·       │
                                       │  EdDSA sign · appcast│
                                       └──────────┬──────────┘
                                                  │ publish
                        ┌─────────────────────────┼─────────────────────────┐
                        ▼                                                     ▼
              ┌───────────────────┐                                ┌───────────────────┐
              │  GitHub Release   │  DMG download (stable URL)      │  codenta.us        │
              │  Uninstally.dmg   │◀───────────────────────────────│  appcast.xml       │
              └───────────────────┘                                └─────────┬─────────┘
                                                                             │ reads only
                                                                             ▼
                                                                   ┌───────────────────┐
                                                                   │  Uninstally.app    │
                                                                   │  Sparkle + EdDSA   │
                                                                   └───────────────────┘
```

## How Sparkle works here

- `UpdateManager` (`Uninstally/ViewModels/UpdateManager.swift`) wraps
  `SPUStandardUpdaterController`. It exposes an observable summary (current/latest
  version, last-checked, status) and the user preferences, and implements
  `SPUUpdaterDelegate` to pin the feed and gate channels.
- The **feed URL is pinned** to `https://codenta.us/appcast.xml` both in
  `Info.plist` (`SUFeedURL`) and in the delegate (`feedURLString(for:)`). No other
  URL is ever read.
- Every update must carry a valid **EdDSA (Ed25519) signature** matching
  `SUPublicEDKey` in `Info.plist`. Unsigned, tampered, or wrong-key updates are
  rejected (fail-closed).
- Sparkle's native UI presents the release notes, download progress (percentage,
  MB, ETA, cancel), and the Install / Later / Skip prompts. This is the
  first-party-quality experience Apple users expect.
- Checks run **on launch** and **every 24 hours** (`SUScheduledCheckInterval`),
  plus **manually** from Settings → Updates → Check Now. Duplicate/concurrent
  checks are prevented.

### Channels

A single appcast serves all channels. Stable items have no channel tag; beta and
nightly items carry `<sparkle:channel>beta</sparkle:channel>` /
`nightly`. `UpdateManager.allowedChannels(for:)` returns an empty set unless the
user enables **Receive Beta Updates**, so stable users never see pre-releases.

## How the appcast works

`appcast.xml` is a Sparkle RSS feed. Each `<item>` describes one release:
`sparkle:version` (build number, used for comparison), `shortVersionString`,
`minimumSystemVersion`, the release notes (`<description>`), and an `<enclosure>`
with the DMG URL, `length`, and `sparkle:edSignature`.

It is generated by `scripts/make_appcast.py`, which **merges** the new item into
the previously published feed (preserving history) and refuses any enclosure URL
that isn't under the official GitHub releases path.

Why host it on `codenta.us` instead of using GitHub Releases directly? Stable,
vendor-controlled URLs: we can move release hosting later (S3, R2, etc.) without
breaking every installed client, because the client only ever knows about
`codenta.us/appcast.xml`.

## How GitHub Actions works

See `.github/workflows/release.yml`. Triggered by any `v*` tag, it runs a **free**
pipeline (no Apple Developer ID, no notarization): version sync → build → test →
DMG → **ad-hoc code sign** → SHA256 → **Sparkle EdDSA sign** → appcast → GitHub
release → website deploy. See `docs/RELEASING.md` for the developer flow and
`docs/SECRETS.md` for the two required secrets.

## Signing & Gatekeeper (no Developer ID)

Update trust does **not** depend on Apple. The security boundary is Sparkle's
**EdDSA signature**, verified against `SUPublicEDKey`. The pipeline **ad-hoc**
signs the app and DMG (`codesign -s -`), which is free and is all that Apple
Silicon requires to *run* a binary.

The only consequence of skipping notarization is the **first manual download**:
macOS Gatekeeper shows an "unidentified developer" prompt once. Two ways to smooth
this over:

- **Homebrew (recommended, no prompt):** `brew install --cask gostonx/tap/uninstally`
- **Direct download:** right-click the app → **Open** once to approve it.

Crucially, once the app is trusted, **Sparkle removes the `com.apple.quarantine`
flag from the updates it installs**, so every subsequent auto-update is prompt-free
— the experience matches a notarized app from then on.


## How website deployment works

The job checks out `gostonx/codenta-site`, writes the regenerated `appcast.xml`,
updates the Uninstally download links and `uninstally-version.json`
(`scripts/sync_website_version.py`), and commits. Cloudflare Pages (connected to
that repo) deploys automatically to `https://codenta.us`. An optional
`wrangler pages deploy` path is included (commented) for direct deploys.

## Key management

### Generate keys (one-time)

```sh
# From the Sparkle tools of the matching version:
./bin/generate_keys
```

- Copy the printed **public** key into `SUPublicEDKey` in
  `Config/Uninstally-Info.plist`.
- Export the **private** key and store it as the `SPARKLE_PRIVATE_KEY` CI secret:
  ```sh
  ./bin/generate_keys -x sparkle_private_key.txt   # then paste + delete
  ```

### Rotating keys

Sparkle supports a smooth rotation because appcasts can be signed and clients
validate against the embedded public key:

1. Generate a **new** key pair.
2. Ship an app update **still signed with the old key** that also embeds the new
   public key in `SUPublicEDKey` (Sparkle allows listing the new key so already
   updated clients trust future signatures).
3. Once the bulk of users are on that version, switch CI (`SPARKLE_PRIVATE_KEY`)
   to sign with the new key.
4. Retire the old key.

### Revoking a compromised key

1. **Immediately** rotate `SPARKLE_PRIVATE_KEY` to a fresh key and update
   `SUPublicEDKey`.
2. Cut a new signed release so honest clients move to the new key.
3. Because Sparkle is fail-closed, any update signed with a key that doesn't match
   the embedded public key is already rejected — an attacker cannot serve updates
   without the private key. Removing the bad items from `appcast.xml` and
   publishing a higher version limits exposure for clients that still trust the
   old key.

## Preservation guarantees

Sparkle replaces the app bundle in place and relaunches. All user state lives in
`UserDefaults` / the app container (Settings, sidebar layout, recent history,
preferences) and is untouched by the update, so nothing is lost across versions.
The app is never left in a broken state: the new bundle is only swapped in after
it is fully downloaded and signature-verified.
